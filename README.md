# X Fraud Scanner (垃圾推号一扫空)

一个浏览器用户脚本（userscript），专门用于在 X（Twitter）推文回复区识别并一键批量 Block 中文诈骗/色情引流账号。可通过 Tampermonkey、Violentmonkey、Greasemonkey 等常见 userscript 管理器安装运行。

---

## 背景

中文 X 用户的推文回复区经常被大量诈骗账号骚扰。这类账号有明显的特征模式：用户名含心形 Emoji、自我介绍包含"真人"/"同城"/"附近的吗"等套词、正文带有夸克/UC 网盘链接，或夹杂"dd"/"搭子"/"固炮"等暗语。这些账号批量注册、批量回复，靠人工逐个 Block 效率极低。

本脚本通过自动扫描当前页面的所有回复，识别符合规则的账号，提供批量 Block 面板和单条 Block 快捷按钮，将整页清理时间从几分钟压缩到几秒。

---

## 安装

1. 安装一个 userscript 管理器，例如 [Tampermonkey](https://www.tampermonkey.net/)、[Violentmonkey](https://violentmonkey.github.io/) 或 [Greasemonkey](https://www.greasespot.net/)
2. 打开脚本文件 `x_fraud_scanner.user.js`，或在 Greasy Fork / UserScripts 页面点击安装
3. userscript 管理器会弹出确认页面，点击"安装"即可
4. 打开任意推文的回复页面（`x.com/*/status/*`），右下角会出现操作按钮

## 功能

### 扫描面板（批量操作）

- **快速扫描按钮（放大镜图标）**：扫描当前已加载的回复，打开面板后自动逐个 Block 命中的账号
- **大杀器！全量扫描按钮（漏斗图标）**：滚动到底部进行完整扫描（最多 60 轮），适合回复数量多的热门推文。扫了一些帖子轻松屏蔽了几千账号，基本上就看不到了。适用于某些热门帖子的回复，一网打尽。
- 面板按列显示所有命中用户；快速扫描会自动开始 Block，全量扫描和列表页批量操作仍可手动确认后执行，完成后该行变灰并划线

### 行内 Block 按钮（逐条操作）

- 每条回复的用户名旁边会注入一个小圆圈图标（禁止符号）
- **红色高亮**：该账号匹配了过滤规则，建议 Block
- **灰色低调**：该账号未命中规则，但仍可点击手动 Block（用于漏报情况）
- Block 成功后图标变为绿色对勾，同页所有该账号的按钮同步更新

### 关键词动态管理

面板顶部工具栏可以在不刷新页面的情况下实时增删关键词：
- **内容关键词**：匹配推文正文
- **用户名关键词**：匹配显示名（Display Name）
- **正则关键词**：匹配推文正文中的固定模式
- **自定义关键词工具**：状态页浮动按钮组里的齿轮按钮支持导出、合并导入和覆盖自定义关键词 JSON；导出只包含手动新增词，不包含系统预设词

### 静音关键词（首页）

在首页（`x.com/home`）出现静音图标按钮：选中页面上的词，点击后复制到剪贴板并跳转到 X 的静音关键词设置页，直接粘贴添加。

## 检测规则

命中以下任一条件即视为嫌疑账号：

| 规则 | 说明 |
|------|------|
| **心形 Emoji 用户名** | 显示名中含 ♥ ❤ 💕 等心形类 Unicode 字符 |
| **用户名关键词** | 显示名包含某些诈骗关键词 （可定制） |
| **正文内容关键词** | 推文正文或卡片预览含某些诈骗关键词等（可定制） |
| **正文正则模式** | 推文正文匹配预设或自定义 JS 正则表达式（可定制） |

### ASCII 短关键词的中文上下文要求

部分关键词是纯英文短词（如 `dd`），在英文推文中可能误报。规则：**若关键词不含中文且英文字母数量不超过 5 个，则要求推文文本中必须同时出现至少一个中文字符，否则跳过该匹配**。这类短词也会启用英文单词边界保护，避免 `dd` 命中 `daddy`。

英文字母数量超过 5 个的 ASCII 关键词或短语（如 `big bro`、`big brother`、`pan.quark.cn`）足够明确，不要求中文上下文，也不做 partial-match 边界限制。

## 技术实现

### Auth Token 获取

X.com 的 Block API 需要两个凭证：Bearer Token 和 CSRF Token（ct0 cookie）。

- **CSRF**：直接从 `document.cookie` 中读取 `ct0` 字段
- **Bearer Token**：脚本在页面启动时 Hook 了 `window.fetch` 和 `XMLHttpRequest.prototype.setRequestHeader`，监听 X.com 自身的 API 请求，从中提取 `Authorization: Bearer ...` 头。捕获到后存入 `liveBearer` 变量。若页面加载后 X.com 尚未发出任何 API 请求，则降级使用内置的公开 Bearer Token（该 Token 是 X.com 客户端通用的，非用户私钥）

### Block API

使用 `GM_xmlhttpRequest`（userscript 管理器提供的跨域请求能力）调用：

```
POST https://x.com/i/api/1.1/blocks/create.json
Authorization: Bearer <token>
x-csrf-token: <ct0>
Content-Type: application/x-www-form-urlencoded

screen_name=<handle>
```

使用 `GM_xmlhttpRequest` 而非页面原生 `fetch` 的原因：避免触发 X.com SPA 的请求拦截逻辑（Block 成功后页面可能自动刷新导致操作中断）。

### X.com 平台限制与速率控制

X.com 对 `blocks/create` 接口没有公开的硬性数字，但反滥用机制会在检测到快速自动化请求时强制登出当前会话（而不是封号）。社区工具（Blue Blocker 等）的实践经验如下：

- **单次会话上限约 500 个**：超过后被登出的概率显著上升，建议当天停止，次日再继续
- **间隔不能太短且不能太均匀**：固定等间隔是强 bot 信号，2023 年后 X 的检测明显变严，此前可用 1 秒/个，现在社区保守标准是 10-15 秒/个
- **本脚本采用的策略**：基础延迟 3000ms 加随机抖动 0-2000ms，实际间隔 3-5 秒，在速度和安全之间取平衡

#### 跨标签页速率共享

多个标签页同时运行时，每个标签页自己的 `sleep()` 只能控制本标签页的节奏，无法防止多标签页并发打满速率。脚本通过以下机制保证跨标签页全局速率：

1. **`navigator.locks`（同源互斥锁）**：所有 `x.com` 标签页共享同一个锁命名空间，`xfs-block-lock` 确保任意时刻只有一个标签页在执行 Block 调用
2. **`localStorage` 时间戳**：每次 Block 成功后写入 `xfs-last-block`，下一次获取锁后先读取该时间戳，再决定还需等待多久才满足最小间隔

两者配合，无论哪个标签页发出了上一次请求，下一次请求都会遵守全局间隔。

### 页面扫描

- 选取 `article[data-testid="tweet"]` 作为推文单元
- 提取用户名时同时处理 `<img alt="...">` 形式的 Emoji（X.com 用 Twemoji 渲染 Emoji 为图片，`textContent` 会丢失，需手动遍历子节点）
- 关键词仅在推文正文（`tweetText`）和卡片预览（`card.wrapper`）范围内匹配，不匹配整个 article（避免把用户主页链接中的字符误判为内容关键词）

### 虚拟滚动处理

X.com 使用虚拟滚动：滚动时旧推文从顶部移除，新推文从底部加入。脚本以"最后一个 article 元素引用是否变化"作为新内容到来的信号（而非计数），并在每轮扫描后将结果合并到累积器（`Map<handle, user>`），确保滚动过的推文不丢失。

### 行内按钮注入

通过 `MutationObserver` 监听 `document.body`，每次 DOM 变化后 debounce 300ms 执行注入。用 `data-xfs-ibtn` 标记已处理的 article，避免重复注入。按钮插入到 caret（三点菜单）节点之前，与用户名同在一个 flex 行中。

### SPA 路由监听

X.com 是单页应用，路由跳转不刷新页面。脚本通过同一个 `MutationObserver` 对比 `location.pathname` 变化，在路由切换时清理旧按钮并在适当延迟后重新注入。

## 已知局限

- Block 后需手动刷新页面，已 Block 的账号才会从回复列表消失（X.com 不会自动重新渲染）
- 脚本会在 `x.com` / `twitter.com` 全站启动以捕获 SPA 路由跳转，但只在推文详情页（`/status/...`）、列表页和首页（`/home`）显示操作按钮
- 全量扫描最多 60 轮，极长的回复串（数千条）可能扫不完
- 心形 Emoji 检测覆盖了常见的 Unicode 心形字符，但不涵盖所有可能的变体
- ASCII 短关键词的中文上下文检测基于整条推文文本，若诈骗账号在同一条推文中完全不出现中文，可能漏报
- **批量 Block 可能触发 X.com 强制登出**：单次会话 Block 数量过多时，X.com 会踢出当前登录状态（不封号）。建议单次会话不超过约 500 个，触发后重新手工登录即可，之后继续操作无碍。脚本已通过 3-5 秒随机间隔降低触发概率，但无法完全规避
